## 1. 项目概述与核心架构

### 1.1 项目定位

面向企业知识库场景，构建一套：

- **MySQL作为主数据源**
- **Elasticsearch作为检索副本**
- **Canal负责监听Binlog**
- **同步服务负责数据写入ES**
- **失败可重试、可补偿、可监控**的知识库全文检索平台

### 1.2 核心架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        Admin 写入链路                            │
│  AdminArticleController → ArticleServiceImpl → MySQL            │
│  AdminCategoryController → CategoryServiceImpl → MySQL            │
│  AdminTagController → TagServiceImpl → MySQL                     │
└─────────────────────────────────────────────────────────────────┘
                              │ Binlog
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        CDC 同步链路                              │
│  CanalServer → CanalSyncListener → CanalEventAdapterImpl         │
│  → SyncEvent → IncrementalSyncServiceImpl.dispatch()            │
│  → ArticleSyncWorker.rebuildAndUpsert()                         │
│  → 回源MySQL → ArticleDocumentBuilder → ES                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        搜索查询链路                              │
│  SearchController → KnowledgeSearchServiceImpl                    │
│  → ArticleSearchQuery → ElasticsearchArticleIndexRepository       │
│  → multi_match(title^3, summary^2, content^1) + filters + sort  │
│  → SearchResponse                                                │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 三条核心链路

| 链路 | 触发方式 | 终点 |
|------|---------|------|
| **Admin写入链路** | HTTP POST/PUT/DELETE | MySQL（业务只写MySQL，不直接操作ES） |
| **CDC同步链路** | Canal监听MySQL binlog | ES文档重建/删除 |
| **搜索查询链路** | HTTP GET /search | ES全文检索 |

### 1.4 关键设计约束

1. **MySQL是唯一事实来源** — 业务写入只落MySQL，ES通过CDC链路派生
2. **回源重建策略** — 不信任binlog字段拼接，而是回源MySQL完整重建ES文档
3. **最终一致性** — 不追求强一致，通过Canal事件驱动+失败补偿实现
4. **REQUIRES_NEW事务** — 失败记录使用独立事务，保证不因外部事务回滚而丢失

## 3. 文章管理功能

**关键文件**: `src/main/java/com/knowledge/search/service/article/ArticleServiceImpl.java`

### 3.1 创建文章

**触发条件**: `POST /admin/article`

**处理步骤**:
1. 参数校验（@Valid + Jakarta Validation）
2. 校验categoryId对应的分类存在（`categoryService.getRequired()`）
3. 校验tagIds对应的标签全部存在（`tagService.getRequiredByIds()`）
4. 构建ArticleEntity，状态固定为DRAFT，isDeleted=false
5. 插入MySQL（自动填充createdAt/updatedAt）
6. 维护文章标签关联（`replaceTags()` — 先删后插）

**数据流转**:
```
ArticleCreateRequest → 校验分类 → 校验标签 → ArticleMapper.insert() → MySQL
                                            ↓
                                      replaceTags() → ArticleTagMapper
```

**核心代码**:
```java
@Transactional
public Long create(ArticleCreateRequest request) {
    CategoryEntity category = categoryService.getRequired(request.categoryId());
    List<TagEntity> tags = tagService.getRequiredByIds(normalizeTagIds(request.tagIds()));

    ArticleEntity entity = new ArticleEntity();
    applyWritableFields(entity, request.title(), request.summary(), request.content(), request.categoryId(), request.author());
    entity.setStatus(ArticleStatus.DRAFT);    // 初始状态固定为草稿
    entity.setIsDeleted(Boolean.FALSE);
    articleMapper.insert(entity);

    replaceTags(entity.getId(), tags);
    return entity.getId();
}
```

### 3.2 更新文章

**触发条件**: `PUT /admin/article/{id}`

**处理步骤**:
1. 校验文章存在且未删除（`getRequiredArticle()`）
2. 校验categoryId存在
3. 校验tagIds存在
4. 更新文章字段
5. 替换标签关联（先删后插）

**注意**: 更新操作通过MySQL触发binlog，Canal会自动捕获并触发ES增量同步。

### 3.3 删除文章（逻辑删除）

**触发条件**: `DELETE /admin/article/{id}`

**处理步骤**:
1. 校验文章存在且未删除
2. 删除文章标签关联记录（`articleTagMapper.deleteByArticleId()`）
3. 执行逻辑删除（`is_deleted = true`）

**核心代码**:
```java
@Transactional
public void delete(Long id) {
    getRequiredArticle(id);
    articleTagMapper.deleteByArticleId(id);
    articleMapper.update(null, new LambdaUpdateWrapper<ArticleEntity>()
            .eq(ArticleEntity::getId, id)
            .eq(ArticleEntity::getIsDeleted, false)
            .set(ArticleEntity::getIsDeleted, true));
}
```

**关键点**: 使用MyBatis-Plus的`@TableLogic`注解实现逻辑删除，DELETE操作会转换为UPDATE。binlog中表现为UPDATE事件，CDC链路会捕获并处理。

### 3.4 发布文章

**触发条件**: `PUT /admin/article/{id}/publish`

**处理步骤**:
1. 校验文章存在且未删除
2. 更新状态为PUBLISHED
3. 设置publishTime（可指定）

**核心代码**:
```java
@Transactional
public void publish(Long id, ArticlePublishRequest request) {
    getRequiredArticle(id);
    articleMapper.update(null, new LambdaUpdateWrapper<ArticleEntity>()
            .eq(ArticleEntity::getId, id)
            .eq(ArticleEntity::getIsDeleted, false)
            .set(ArticleEntity::getStatus, ArticleStatus.PUBLISHED)
            .set(ArticleEntity::getPublishTime, request.publishTime()));
}
```

**状态机**:
```
DRAFT ──publish()──> PUBLISHED
PUBLISHED ──offline()──> OFFLINE
OFFLINE ──publish()──> PUBLISHED（可重新设置publishTime）
```

**关键点**: 发布后binlog UPDATE事件触发CDC链路，`ArticleSyncWorker.rebuildAndUpsert()`检查文章状态=PUBLISHED才构建ES文档，否则执行删除。

### 3.5 下线文章

**触发条件**: `PUT /admin/article/{id}/offline`

**处理步骤**:
1. 校验文章存在且未删除
2. 更新状态为OFFLINE

**关键点**: 下线后文章仍存在于ES索引中，但搜索默认只返回status=PUBLISHED的文章，所以下线后会从公开搜索结果中消失。

### 3.6 文章分页查询

**触发条件**: `GET /admin/article`

**查询参数**:
- `keyword`: 关键字（模糊匹配title/summary/content）
- `status`: 状态过滤
- `categoryId`: 分类ID过滤
- `pageNum` / `pageSize`: 分页参数

**处理步骤**:
1. 构建MyBatis-Plus查询条件（自动拼接is_deleted=false）
2. 分页查询文章列表
3. 批量查询分类信息（按categoryId去重）
4. 批量查询标签信息（`findTagsByArticleIds()`避免N+1）
5. 组装响应返回

**批量查询优化**:
```java
// 批量查询标签，按articleId分组
private Map<Long, List<TagEntity>> findTagsByArticleIds(Collection<Long> articleIds) {
    List<ArticleTagEntity> relations = articleTagMapper.selectByArticleIds(articleIds);
    List<TagEntity> tags = tagService.getRequiredByIds(relations.stream().map(ArticleTagEntity::getTagId).toList());
    // 按articleId分组，标签按relation id排序
}
```

```java
public List<TagEntity> getRequiredByIds(Collection<Long> ids) {
    if (ids == null || ids.isEmpty()) return List.of();
    List<TagEntity> tags = tagMapper.selectBatchIds(ids);
    if (tags.size() != ids.stream().distinct().count()) {
        throw new BusinessException(ErrorCode.TAG_NOT_FOUND, "One or more tags do not exist");
    }
    return tags;
}
```

### 6.3 多字段匹配与权重

```java
// 搜索策略：multi_match，字段权重 title^3 > summary^2 > content^1
mustQueries.add(QueryBuilders.multiMatch()
        .query(query.keyword())
        .fields("title^3", "summary^2", "content")
        .type(TextQueryType.BestFields)
        .build()._toQuery());
```

**权重含义**:
- 标题匹配：相关性得分×3
- 摘要匹配：相关性得分×2
- 正文匹配：相关性得分×1

### 6.4 过滤条件构建

所有过滤条件放入`filterQueries`（不影响评分）:

```java
// 分类过滤（精确term）
if (query.categoryId() != null) {
    filterQueries.add(QueryBuilders.term().field("categoryId").value(query.categoryId()).build()._toQuery());
}

// 状态过滤（精确term）
if (StringUtils.hasText(query.status())) {
    filterQueries.add(QueryBuilders.term().field("status").value(query.status()).build()._toQuery());
}

// 作者过滤（精确term）
if (StringUtils.hasText(query.author())) {
    filterQueries.add(QueryBuilders.term().field("author").value(query.author()).build()._toQuery());
}

// 标签过滤（AND语义 — 每个标签名单独term查询，全部匹配才算通过）
if (!CollectionUtils.isEmpty(query.tagNames())) {
    query.tagNames().stream()
            .filter(StringUtils::hasText)
            .map(tagName -> QueryBuilders.term().field("tagNames").value(tagName).build()._toQuery())
            .forEach(filterQueries::add);
}

// 时间范围过滤
addRangeFilter(filterQueries, "publishTime", query.publishTimeStart(), query.publishTimeEnd());
addRangeFilter(filterQueries, "updatedAt", query.updatedTimeStart(), query.updatedTimeEnd());
```

### 6.5 排序规则

```java
private List<SortOptions> buildSorts(String keyword, String sort) {
    List<SortOptions> sorts = new ArrayList<>();
    // 有关键字且排序≠latest时，按ES相关度评分降序
    if (!"latest".equalsIgnoreCase(sort) && StringUtils.hasText(keyword)) {
        sorts.add(SortOptions.of(option -> option.score(score -> score.order(SortOrder.Desc))));
    }
    // 默认排序：publishTime DESC → updatedAt DESC → id DESC
    sorts.add(SortOptions.of(option -> option.field(field -> field.field("publishTime").order(SortOrder.Desc))));
    sorts.add(SortOptions.of(option -> option.field(field -> field.field("updatedAt").order(SortOrder.Desc))));
    sorts.add(SortOptions.of(option -> option.field(field -> field.field("id").order(SortOrder.Desc))));
    return sorts;
}
```

### 6.6 高亮规则

```java
private HighlightQuery buildHighlightQuery() {
    Highlight highlight = new Highlight(
            HighlightParameters.builder()
                    .withPreTags("<em>")
                    .withPostTags("</em>")
                    .build(),
            List.of(
                    new HighlightField("title"),
                    new HighlightField("summary"),
                    new HighlightField("content")));
    return new HighlightQuery(highlight, ArticleDocument.class);
}
```

**高亮策略**:
- 高亮标签：`<em>关键字</em>`
- 高亮字段：title、summary、content
- 只返回第一个匹配片段
- **仅在keyword非空时启用高亮**

### 6.7 搜索默认行为

- **默认只搜索已发布文章**（status默认=PUBLISHED）
- **默认启用高亮**（highlight默认=true）
- **默认按相关度排序**（sort默认=relevance，但无关键字时自动降级为latest）

### 7.1 整体流程

```
MySQL binlog
    │
    ▼
CanalServer
    │
    ▼ (getWithoutAck拉取)
CanalSyncListener.pollLoop()
    │
    ▼ handleEntries() 解析ROWDATA
    │
    ▼ CanalMessage (schema/table/eventType/beforeColumns/afterColumns)
    │
    ▼ CanalEventAdapterImpl.adapt()
    │
    ▼ SyncEvent { bizType, bizId, operation, payload }
    │
    ▼ IncrementalSyncServiceImpl.handle()
    │
    ▼ dispatch() 按bizType分发
    │
    ├── BizType.ARTICLE ──→ handleArticle()
    │                        ├── DELETE → articleSyncWorker.delete()
    │                        └── REBUILD → articleSyncWorker.rebuildAndUpsert()

    │

    │

SyncFailureService.recordFailure() → sync_fail_log表
```

### 7.4 增量同步分发

```java
private void dispatch(SyncEvent event, boolean recordFailure) {
    switch (event.bizType()) {
        case ARTICLE -> executeSafely(event, () -> handleArticle(event), recordFailure);
        case CATEGORY -> rebuildArticles(articleMapper.selectIdsByCategoryId(event.bizId()), event.bizType(), recordFailure);
        case TAG -> rebuildArticles(articleTagMapper.selectArticleIdsByTagId(event.bizId()), event.bizType(), recordFailure);
        case ARTICLE_TAG_RELATION -> rebuildArticles(List.of(event.bizId()), event.bizType(), recordFailure);
    }
}

private void handleArticle(SyncEvent event) {
    if (event.operation() == SyncOperation.DELETE) {
        articleSyncWorker.delete(event.bizId());  // 直接从ES删除
        return;
    }
    articleSyncWorker.rebuildAndUpsert(event.bizId());  // 重建ES文档
}
```

**CATEGORY/TAG变更处理策略**: 不是更新关联表本身，而是找到受影响的文章，再触发文章维度的ES文档重建。例如：
- 分类名变更 → 查出该分类下所有文章ID → 逐篇重建ES文档
- 标签名变更 → 查出所有引用该标签的文章ID → 逐篇重建ES文档

### 7.5 文章同步Worker

```java
public void rebuildAndUpsert(Long articleId) {
    // 从MySQL回源构建文档
    var document = articleDocumentBuilder.buildByArticleId(articleId);
    if (document == null) {
        // 文章不存在或非PUBLISHED状态 → 从ES删除
        articleIndexRepository.deleteById(articleId);
        return;
    }
    // 文档存在且已发布 → 写入ES
    articleIndexRepository.save(document);
}
```

**关键点**: `buildByArticleId()`内部会检查文章是否`is_deleted=false`且`status=PUBLISHED`，如果不满足则返回null，导致ES删除操作。

### 7.6 ES文档构建（回源MySQL）

**设计原则**: 不信任binlog中的部分字段，而是回源MySQL读取完整快照后重建文档。

```java
public List<ArticleDocument> buildByArticles(Collection<ArticleEntity> articles) {
    // 1. 过滤：只保留未删除 + 状态=PUBLISHED 的文章
    List<ArticleEntity> eligibleArticles = articles.stream()
            .filter(article -> article != null && !Boolean.TRUE.equals(article.getIsDeleted()))
            .filter(article -> article.getStatus() == ArticleStatus.PUBLISHED)
            .toList();
    if (eligibleArticles.isEmpty()) return List.of();

    // 2. 批量查询关联数据（避免N+1）
    Map<Long, CategoryEntity> categoriesById = ...;  // selectBatchIds
    Map<Long, List<Long>> tagIdsByArticleId = ...;    // selectByArticleIds
    Map<Long, TagEntity> tagsById = ...;             // selectBatchIds

    // 3. 组装文档
    return eligibleArticles.stream()
            .map(article -> toDocument(article, categoriesById, tagIdsByArticleId, tagsById))
            .toList();
}
```

**批量查询优化**:
- `categoryMapper.selectBatchIds()` — 一次性查出所有分类
- `articleTagMapper.selectByArticleIds()` — 批量查出所有文章-标签关联
- `tagMapper.selectBatchIds()` — 一次性查出所有标签详情
- 在内存中按articleId分组拼装，避免逐条查询

### 8.2 全量同步流程

```java
public String triggerFullSync() {
    articleIndexRepository.deleteAll();  // 1. 清空ES索引（重建式全量同步）
    long current = 1L;
    do {
        // 2. 分页查询MySQL（is_deleted=false AND status=PUBLISHED）
        Page<ArticleEntity> page = articleMapper.selectPage(
                new Page<>(current, appSyncProperties.full().pageSize()),
                new LambdaQueryWrapper<ArticleEntity>()
                        .eq(ArticleEntity::getIsDeleted, false)
                        .eq(ArticleEntity::getStatus, ArticleStatus.PUBLISHED)
                        .orderByAsc(ArticleEntity::getId));
        // 3. 同步这一页
        synced += syncPage(page.getRecords());
        current++;
    } while (current <= page.getPages());
}
```

### 8.3 批量同步与降级

```java
private long syncPage(List<ArticleEntity> articles) {
    try {
        // 批量构建 + 批量写入ES
        List<ArticleDocument> documents = articleDocumentBuilder.buildByArticles(articles);
        articleIndexRepository.saveAll(documents);
        return documents.size();
    } catch (Exception batchException) {
        // 批量失败，降级为逐条处理
        for (ArticleEntity article : articles) {
            try {
                articleSyncWorker.rebuildAndUpsert(article.getId());
                synced++;
            } catch (Exception exception) {
                // 单条失败也记入sync_fail_log
                syncFailureService.recordFailure(
                        new SyncEvent(BizType.ARTICLE, article.getId(), SyncOperation.REBUILD, "FULL_SYNC"),
                        exception);
            }
        }
    }
}
```

**降级策略**:
- 整页批量失败 → 逐条重建 + 逐条写入
- 单条失败 → 记录到`sync_fail_log`，不影响其他记录
- 保留失败可追溯能力

### 9.1 失败记录

**触发时机**:
- `IncrementalSyncServiceImpl.executeSafely()` — 捕获同步异常时
- `FullSyncServiceImpl.syncPage()`单条降级时失败

**REQUIRES_NEW事务保证**:

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void recordFailure(SyncEvent event, Exception exception) {
    // 使用独立事务，保证不因外部事务回滚而丢失
    SyncFailLogEntity entity = new SyncFailLogEntity();
    entity.setBizType(event.bizType());
    entity.setBizId(event.bizId());
    entity.setPayload(toPayload(event));
    entity.setErrorMsg(exception.getMessage());
    entity.setRetryCount(0);
    entity.setStatus(SyncFailStatus.PENDING);
    entity.setNextRetryTime(LocalDateTime.now().plus(appSyncProperties.retry().initialDelay()));
    syncFailLogMapper.insert(entity);
}
```

### 9.2 定时重试任务

```java
@Scheduled(fixedDelayString = "${app.sync.retry.scan-interval:PT5M}")  // 默认5分钟
public void scanAndRetry() {
    if (!appSyncProperties.retry().enabled()) return;
    syncFailureService.retryPendingFailures();
}
```

**扫描条件**: `status = PENDING AND next_retry_time <= NOW()`

### 9.3 重试策略

**配置参数**:

```yaml
app:
  sync:
    retry:
      enabled: true
      max-attempts: 3          # 最多重试3次
      initial-delay: PT1M       # 失败后1分钟可重试
      scan-interval: PT5M       # 每5分钟扫描一次
```

**重试状态变更**:

```java
private void retryEntity(SyncFailLogEntity entity) {
    try {
        // 调用handleWithoutFailureRecord避免重试时重复记录失败
        incrementalSyncService.handleWithoutFailureRecord(event);
        // 成功：标记SUCCESS
        syncFailLogMapper.update(...).set(SyncFailLogEntity::getStatus, SyncFailStatus.SUCCESS);
    } catch (Exception exception) {
        int nextRetryCount = entity.getRetryCount() + 1;
        boolean exhausted = nextRetryCount >= appSyncProperties.retry().maxAttempts();
        if (exhausted) {
            // 达到最大次数：标记FAILED，不再重试
            .set(SyncFailLogEntity::getStatus, SyncFailStatus.FAILED)
            .set(SyncFailLogEntity::getNextRetryTime, null);
        } else {
            // 未达最大次数：状态保持PENDING，nextRetryTime往后推1分钟
            .set(SyncFailLogEntity::getStatus, SyncFailStatus.PENDING)
            .set(SyncFailLogEntity::getNextRetryTime, LocalDateTime.now().plus(initialDelay));
        }
    }
}
```

**关键特性**:
- **固定间隔退避** — 每次失败都是1分钟后重试，不指数增长
- **有限次重试** — 3次后标记FAILED，需人工介入
- **独立事务** — 重试时调用`handleWithoutFailureRecord()`避免循环记录失败
