# 2026.4.15-项目架构

## 1. 项目定位

本项目是一个面向企业知识库场景的检索与 CDC 同步平台，当前默认目标是构建一条稳定的 V1 主链路：

`后台管理写入 MySQL -> Canal 订阅 binlog -> 同步服务分发事件 -> Sync Worker 回源 MySQL 重建 ES 文档 -> Elasticsearch 提供搜索`

当前架构的核心原则是：

- MySQL 是唯一事实来源
- Elasticsearch 是派生搜索副本，不承载业务写入真源职责
- 业务写接口不直接双写 MySQL 和 Elasticsearch
- 增量同步采用事件触发 + MySQL 回源重建，而不是按 binlog 字段直接 patch ES
- 同步失败必须可记录、可重试、可人工介入

这意味着项目的重点不是“把 ES 接上”，而是围绕最终一致性建立一套可运行、可补偿、可维护的同步闭环。

---

## 2. 当前 V1 范围

截至当前代码状态，V1 已覆盖以下能力：

- 文章 CRUD、发布、下线
- 分类管理、标签管理
- 基于 Elasticsearch 的全文搜索
- 标题 / 摘要 / 正文多字段检索
- 分类、标签、状态、作者、发布时间范围、更新时间范围过滤
- 高亮返回
- 全量同步
- 基于 Canal 的增量同步
- 同步失败日志记录
- 手动重试、定时重试
- 基础同步统计查询

当前不属于默认已实现范围的内容包括：

- RabbitMQ 解耦消费
- Redis 缓存
- IK 分词改造
- 搜索建议、热词统计
- 完整监控面板
- Prometheus / Grafana 指标体系

这些能力可以作为后续增强，但不应混淆为当前 V1 已落地事实。

## 3. 技术栈与模块分层

### 3.1 技术栈

- Spring Boot 3.3.7
- Java 21
- MyBatis-Plus 3.5.8
- MySQL
- Elasticsearch
- Canal
- Knife4j / Springdoc OpenAPI
- Spring Boot Test + H2

### 3.2 代码分层

项目采用单体分层结构，但职责边界相对清晰：

1. `controller`
   - 对外暴露管理端 API 和搜索 API
   - 管理端包含文章、分类、标签、同步管理接口
   - 搜索端负责公开检索请求

2. `service`
   - 承担业务编排
   - 文章、分类、标签服务负责 MySQL 业务处理
   - 搜索服务负责组装查询条件并委托索引仓储
   - 同步服务负责全量同步、增量分发、失败记录与重试

3. `mapper`
   - 基于 MyBatis-Plus 访问 MySQL
   - 持久化与查询逻辑集中在这一层

4. `search`
   - 定义 ES 文档模型、文档构建器、搜索查询对象、索引仓储抽象
   - 隔离搜索实现细节，避免业务层直接依赖 ES API

5. `sync`
   - `canal`：消费 binlog 事件并适配为内部 `SyncEvent`
   - `worker`：执行按文章维度的重建 / 删除
   - `retry`：定时扫描失败记录并做补偿重试

6. `common` / `config`
   - 通用响应、异常、枚举、配置绑定、OpenAPI 配置

## 5. 三条核心链路

### 5.1 管理写入链路

管理端接口负责文章、分类、标签的写入，但业务写路径只落 MySQL，不直接写 Elasticsearch。

以文章为例：

1. `AdminArticleController` 接收请求
2. `ArticleServiceImpl` 做参数校验、分类 / 标签校验、文章状态流转和关系表维护
3. `ArticleMapper`、`ArticleTagMapper` 完成 MySQL 持久化

这一层不做 ES 双写，避免把业务写入和搜索索引更新耦合在一个同步事务中，从架构上规避常见的双写一致性问题。

### 5.2 搜索查询链路

当前搜索链路是：

`SearchController -> KnowledgeSearchServiceImpl -> ArticleIndexRepository`

搜索副本来自 Elasticsearch，而不是直接查 MySQL。当前实现已经具备以下特征：

- 标题、摘要、正文多字段检索
- 相关度优先，兼顾时间排序
- 支持分类、标签、状态、作者、发布时间范围、更新时间范围过滤
- 使用 Elasticsearch 原生 highlight 返回高亮字段
- 对外公开搜索默认只返回 `published` 文章

当前搜索策略与代码实现是一致的，不再是早期基于字符串替换的演示型高亮逻辑。

### 5.3 CDC 增量同步与补偿链路

当前增量同步链路已经是可运行实现，而不是占位代码：

1. `CanalSyncListener` 持续从 Canal 拉取消息
2. `CanalEventAdapterImpl` 将表变更转换为统一 `SyncEvent`
3. `IncrementalSyncServiceImpl` 根据 `bizType` 和 `operation` 分发处理
4. `ArticleSyncWorker` 负责重建并写入 ES，或删除 ES 文档
5. 发生异常时，由 `SyncFailureServiceImpl` 写入 `sync_fail_log`
6. `SyncRetryJob` 定时扫描待重试记录并再次触发补偿

当前监听的核心表包括：

- `kb_article`
- `kb_category`
- `kb_tag`
- `kb_article_tag`

不同表变更最终都会归约到“找到受影响文章，再按文章维度重建 ES 文档”的思路上，这符合知识库搜索索引是聚合结果而不是单表投影的架构定位。

## 6. 文档重建与索引更新策略

`ArticleDocumentBuilderImpl` 是同步质量的关键组件。它不是根据 binlog 里的局部字段直接修改 ES，而是按当前 MySQL 快照重建文章文档。

其核心逻辑是：

1. 读取文章主记录
2. 校验文章是否未删除且已发布
3. 加载分类信息
4. 加载文章与标签关系
5. 加载标签明细
6. 组装 `ArticleDocument`

这一策略保证了以下几件事：

- 分类名称修改可以通过文章重建传导到 ES
- 标签名称变化、标签关系变化都能回收敛成完整文档重建
- ES 中存的是当前聚合快照，而不是多个 patch 累积的脆弱结果

这也是本项目在一致性设计上的关键取舍。

## 7. 全量同步实现现状

当前 `FullSyncServiceImpl` 已经不是逐条重建 + 逐条写入的低效实现，而是做过一轮性能优化后的版本。

当前全量同步流程是：

1. 校验 `app.sync.full.enabled`
2. 清空现有索引
3. 分页查询 MySQL 中“已发布、未删除”的文章
4. 按页批量调用 `ArticleDocumentBuilder.buildByArticles(...)`
5. 按页批量调用 `ArticleIndexRepository.saveAll(...)`
6. 如果整页批处理失败，则降级到逐条重建
7. 单条失败写入 `sync_fail_log`

这版实现的意义在于：

- 保留 V1 边界，不改变主架构
- 减少 N+1 查询和 ES 单条写入开销
- 兼顾吞吐和稳定性
- 仍然保留失败记录与补偿能力

因此，全量同步当前既是索引初始化能力，也是一个可重复执行的补偿入口。

## 8. 失败补偿与运维能力

当前项目已经具备基础可运维能力，主要体现在同步失败处理链路上。

### 8.1 已实现能力

- `sync_fail_log` 独立记录失败事件
- 失败记录包含 `bizType`、`bizId`、`payload`、错误信息、重试次数、下次重试时间等信息
- 支持手动重试：`POST /admin/sync/retry/{failId}`
- 支持定时重试：`SyncRetryJob`
- 支持失败记录分页查询：`GET /admin/sync/fail/list`
- 支持基础统计查询：`GET /admin/sync/stats`

### 8.2 当前统计能力边界

当前 `/admin/sync/stats` 主要提供的是失败 / 重试状态的汇总视图，适合做基础运维查询，但还不是完整的监控面板能力。

当前尚未落地的能力包括：

- Prometheus 指标暴露
- 全量 / 增量耗时指标
- Canal 消费时序指标
- 重试成功率时序图
- 独立监控面板

因此，现阶段应把“同步指标与监控面板”视为下一步架构增强，而不是已完成能力。

## 9. 当前已验证的架构结论

结合当前代码、测试与已有联调文档，可以明确给出以下结论：

### 9.1 V1 主链路已跑通

根据 `2026.4.15-V1链路测试结论.md`，当前已验证：

- 管理端写 MySQL 可用
- Canal 能监听到 binlog 变化
- 增量同步服务可以正确分发事件
- `ArticleSyncWorker` 能基于 MySQL 快照重建 ES 文档
- 搜索接口能读到同步后的最新索引结果
- 文章下线后可从公开搜索结果中移除
- 分类名、标签名变化可以驱动关联文章重建

### 9.2 搜索能力已不是初版演示形态

根据当前代码与 `2026.4.15-搜索改进.md`，搜索侧已经完成：

- 原生高亮接入
- 更多查询过滤条件
- 默认只检索公开内容
- 更符合实际体验的排序策略

### 9.3 全量同步做过性能增强

根据当前代码与 `2026.4.15-全量同步优化.md`，全量同步已具备：

- 分页处理
- 批量文档构建
- 批量 ES 写入
- 单条兜底回退

因此，当前项目不是“只写了架构图”，而是已经形成了可联调、可验证、可补偿的 V1 实现。

## 10. 当前架构特点总结

如果用一句话概括当前系统，可以表述为：

这是一个以 MySQL 为事实源、以 Canal 为 CDC 入口、以 MySQL 快照重建 Elasticsearch 聚合文档、并内置失败补偿闭环的知识库搜索系统。

它当前最有价值的地方，不是技术栈是否复杂，而是以下几点已经基本成立：

1. 管理写入与搜索索引更新职责解耦
2. 增量同步采用事件触发 + 回源重建，而不是脆弱双写
3. 分类、标签等关联数据变化可以正确传导到聚合文档
4. 失败同步具备记录、重试、补偿闭环
5. 全量同步、增量同步、公开搜索三条链路彼此衔接完整

## 11. 当前缺口与下一步演进

在不推翻现有 V1 架构的前提下，下一步更合理的演进方向包括：

### 11.1 同步指标与监控面板

优先补充同步链路的观测能力，包括：

- 全量同步总耗时
- 分页构建耗时
- ES 批量写入耗时
- 增量事件成功数 / 失败数
- 当前待重试积压
- 重试成功率

建议方向是：

- 应用内埋点
- 预留 Prometheus 暴露能力
- 管理端聚合监控接口
- 后续再接 Grafana 或内置后台页

### 11.2 失败日志上下文增强

当前失败日志已可用，但还可以补强：

- 来源模块
- 失败时间区间
- 最近一次重试时间
- 错误类型
- 更统一的结构化日志字段

### 11.3 索引演进策略

若后续涉及 mapping 或 analyzer 调整，建议采用：

- 新建版本化索引
- 全量重建
- 别名切换

而不是直接覆盖线上索引。

### 11.4 单元测试与审计能力增强

可以继续加强：

- `ArticleDocumentBuilderImpl`
- `IncrementalSyncServiceImpl`
- `SyncFailureServiceImpl`
- ES 查询组装逻辑

并逐步补齐更稳定的同步审计视角。

---

## 12. 结论

就当前 V1 范围而言，这个项目已经不是简单的 CRUD + 搜索集成，而是具备了较完整的“业务数据 -> 搜索副本 -> 故障补偿”闭环。

现阶段最合理的路线不是推翻重做，而是在保持现有架构边界稳定的前提下，继续把它从“链路已跑通”推进到“可观测、可维护、更接近生产系统”。
