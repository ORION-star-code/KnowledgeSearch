# 2026.4.15 V1 链路测试结论

## 1. 结论

截至 2026 年 4 月 15 日，项目 V1 默认主链路已经跑通。

当前已验证通过的链路为：

`后台管理写 MySQL -> Canal 订阅 binlog -> 增量同步服务 -> ArticleSyncWorker 重建文档 -> Elasticsearch 写入/删除 -> 搜索接口返回结果`

这说明项目当前具备以下能力：

- 业务数据先落 MySQL
- Canal 能监听到 MySQL binlog 变更
- 增量同步服务能正确分发同步事件
- Sync Worker 能基于 MySQL 快照重建 ES 文档
- Elasticsearch 检索副本会随业务状态变化更新
- 搜索接口可以读取到增量同步后的最新索引结果

---

## 2. 本次验证环境

本次验证基于本机已启动环境进行。

端口连通性检查结果：

- 应用服务 `127.0.0.1:8080` 正常
- Canal 服务 `127.0.0.1:11111` 正常
- MySQL 服务 `127.0.0.1:3307` 正常
- Elasticsearch 服务 `127.0.0.1:9200` 正常

说明 V1 所依赖的核心组件都处于可联调状态。

---

## 3. 已验证通过的场景

### 3.1 文章发布后的增量同步

验证步骤：

1. 通过管理端接口创建分类、标签、文章
2. 调用文章发布接口
3. 不触发手动全量同步
4. 直接轮询搜索接口

验证结果：

- 新发布文章可以被搜索接口查到
- 说明 Canal 增量同步链路生效
- 说明 ES 文档已由增量同步自动建立

本次验证样例：

- `ArticleId=9`
- `Title=CanalE2E-20260415132405`
- 搜索结果命中成功

### 3.2 文章下线后的索引移除

验证步骤：

1. 对已发布文章调用下线接口
2. 轮询搜索接口查询该文章

验证结果：

- 下线后文章不再出现在 `status=published` 的搜索结果中
- 说明文章状态变化已通过 Canal 驱动 ES 副本更新
- 说明“已下线内容从检索副本剔除”的逻辑正常

本次验证样例：

- `ArticleId=9`
- 下线后搜索结果 `Total=0`

### 3.3 分类名称变更后的关联文章重建

验证步骤：

1. 创建分类、标签、文章并发布
2. 确认文章已进入 ES
3. 修改分类名称
4. 轮询搜索接口查看文章中的 `categoryName`

验证结果：

- ES 中关联文章的分类名称随分类表变更而更新
- 说明 Canal 对分类表变更的监听正常
- 说明系统会根据分类变更找到受影响文章并触发重建

### 3.4 标签名称变更后的关联文章重建

验证步骤：

1. 创建标签并与文章关联
2. 发布文章并确认已被索引
3. 修改标签名称
4. 轮询搜索接口查看文章中的 `tagNames`

验证结果：

- ES 中关联文章的标签名称随标签表变更而更新
- 说明标签维度的聚合文档重建逻辑正常

本次验证样例：

- `ArticleId=11`
- `CategoryName=CategoryNew20260415132544`
- `TagName=TagNew20260415132544`

## 4. 本次验证覆盖到的 V1 能力

本次实际联调已经覆盖了以下 V1 关键能力：

- 管理端写 MySQL
- Canal 订阅 binlog
- 文章发布后的增量索引建立
- 文章下线后的增量索引删除
- 分类变更触发关联文章重建
- 标签变更触发关联文章重建
- ES 检索副本读取
- 搜索接口结果校验

从主链路角度看，V1 已经不是“代码上写了流程”，而是“真实联调已跑通”。

## 5. 暂未专门验证的场景

虽然主链路已经通过，但以下场景本次没有逐条专项验证：

- 文章删除事件是否正确触发 ES 删除
- `kb_article_tag` 关系表新增/删除时是否正确触发文章重建
- Canal 异常断连后的自动恢复表现
- 同步失败记录是否在真实 Canal 触发失败时正确入库
- 定时重试任务是否能把失败同步补偿成功
- 全量同步与增量同步混合执行时的行为稳定性

这些属于后续补充验证项，不影响“V1 主链路已跑通”的结论。

---

## 6. 本次测试中的注意点

本次验证中出现过一次误判：

- 使用中文分类名做精确比对时，PowerShell 控制台输出存在编码显示问题
- 搜索接口实际已经返回数据，但终端展示成了乱码或问号

后续做自动化联调或脚本化验收时，建议：

- 优先使用 ASCII 测试数据
- 或把结果直接落文件后再校验编码

这属于终端显示问题，不是 Canal 同步故障。

---

## 7. 最终判断

当前可以明确给出的判断是：

1. V1 默认架构链路已经跑通
2. Canal 在当前环境中确实参与了增量同步
3. 项目已经具备“写 MySQL、增量同步 ES、搜索读取 ES”的闭环能力
4. 分类和标签等关联数据变化也能驱动文章索引重建

因此，当前项目可以认为：

`V1 主流程可用，核心 CDC 增量同步闭环成立。`
