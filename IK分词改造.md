# IK分词改造

## 1. 背景与目标

当前项目的搜索链路已经具备完整的 V1 架构闭环：

- MySQL 作为唯一事实源
- Canal 负责增量变更捕获
- Sync Worker 负责回源 MySQL 重建 ES 文档
- Elasticsearch 负责搜索查询

目前搜索字段虽然已经支持标题、摘要、正文多字段检索，但 Elasticsearch 映射仍使用默认 `text` 类型，没有针对中文分词做专门配置。这会导致中文检索体验存在明显不足：

- 中文关键词切分不稳定
- 标题、摘要、正文的中文召回率不高
- 用户输入短语或组合词时，结果相关性不足

本次 IK 分词改造的目标是：

1. 为文章搜索字段接入 IK 中文分词
2. 在不破坏现有 MySQL -> 重建 -> ES 架构的前提下完成索引升级
3. 采用低风险的版本化索引切换方案，而不是直接覆盖现有索引
4. 为后续正式开发提供明确、可执行的实施基线

---

## 2. 当前现状

### 2.1 现有文章索引模型

当前 `ArticleDocument` 中：

- `title`、`summary`、`content` 为 `FieldType.Text`
- `categoryName` 也是 `FieldType.Text`
- `tagNames`、`author`、`status` 为 `Keyword`

但这些 `Text` 字段没有显式指定 analyzer，也没有自定义索引 settings。

### 2.2 现有索引初始化方式

当前 `ElasticsearchArticleIndexRepository.deleteAll()` 的行为是：

1. 删除索引
2. 重新创建索引
3. 调用 `putMapping()`

这个流程适合开发或测试环境快速重建，但不适合接入 IK 后的正式升级，因为：

- analyzer 变化本质上会影响索引结构
- 不能在原有索引上直接安全修改
- 线上直接删索引重建有明显风险

### 2.3 现有查询方式

当前搜索查询通过 `multi_match` 在以下字段检索：

- `title^3`
- `summary^2`
- `content`

这个多字段权重策略本身是合理的，接入 IK 后可以继续沿用，不需要推翻当前搜索接口设计。

### 2.4 现有同步方式

当前全量和增量同步都遵循既有架构边界：

- 全量同步由 `FullSyncServiceImpl` 拉取 MySQL 已发布文章重建 ES 文档
- 增量同步由 Canal 事件触发回源重建
- 业务写入不直接双写 MySQL 和 ES

这个模式与 IK 改造不冲突，应保持不变。

---

## 3. 接入前置条件

IK 分词器是 Elasticsearch 节点插件，不是 Java 依赖包。

因此，正式接入前必须满足以下条件：

### 3.1 插件安装

在每个 Elasticsearch 节点安装与当前 ES 版本严格匹配的 `analysis-ik` 插件。

要求：

- 所有数据节点都安装
- 插件版本与 ES 版本一致
- 安装后重启节点

### 3.2 插件校验

在应用创建新索引前，先确认节点确实已加载 IK 插件。

建议校验方式：

- 通过 `_cat/plugins`
- 或通过节点插件信息接口

若节点未安装 IK，不应继续执行索引创建或切换。

### 3.3 职责边界

插件安装属于部署/运维职责，不放在 Java 服务中自动执行。

应用层只负责：

- 创建带 IK 的索引 settings 和 mappings
- 使用新索引执行全量构建
- 切换查询别名

---

## 4. 索引改造方案

### 4.1 不直接修改现有索引

现有索引不做原地 mapping/analyzer 修改。

原因：

- analyzer 改动通常要求重建索引
- 原地修改风险高
- 会影响现有在线查询

因此本次方案采用：

- 新建版本化索引
- 全量重建数据
- 别名切换
- 保留旧索引作为回滚点

### 4.2 版本化索引命名

建议索引命名方式：

- 旧索引：`kb_article_index`
- 新索引：`kb_article_index_v2`
- 查询/写入别名：`kb_article_index_current`

后续如果继续演进，可扩展为：

- `kb_article_index_v3`
- `kb_article_index_v4`

### 4.3 别名策略

系统不直接固定使用物理索引名，而是统一通过别名访问活动索引。

建议：

- 查询读别名
- 全量构建写目标新索引
- 切换完成后把别名指向新索引

这样可以保证：

- 应用代码无需随每次升级修改索引名
- 切换和回滚更容易控制

---

## 5. 字段与分词策略

### 5.1 需要接入 IK 的字段

本次建议对以下字段接入 IK：

- `title`
- `summary`
- `content`
- 可选：`categoryName`

其中核心字段是前三个，`categoryName` 是否使用 IK 可按后续搜索需求决定。

### 5.2 推荐 analyzer 组合

建议采用常见且稳定的组合：

- 建索引时使用 `ik_max_word`
- 搜索时使用 `ik_smart`

原因：

- `ik_max_word` 召回更充分
- `ik_smart` 查询更收敛，噪音更低
- 这是中文检索里比较常用的保守组合

### 5.3 精确字段保留策略

需要精确匹配、过滤或排序的字段，不能全部改成分词字段。

因此：

- `tagNames`、`author`、`status` 继续保留 `keyword`
- 对需要同时支持全文搜索和精确能力的文本字段，可增加 `keyword` 子字段

例如：

- `title` 主字段用于 IK 分词检索
- `title.keyword` 用于精确匹配或排序

## 6. 应用层改造点

### 6.1 ArticleDocument

`ArticleDocument` 需要从“仅靠默认注解建模”升级为“与明确索引 mapping 一致”。

建议做法：

- 不是只改注解后继续依赖默认 `putMapping()`
- 而是由索引创建逻辑显式提交 settings 和 mappings

这样更适合：

- 控制 analyzer
- 控制字段子字段结构
- 后续继续演进 synonym / 拼音 / 自定义 analyzer

### 6.2 ElasticsearchArticleIndexRepository

这里需要新增或改造的能力包括：

- 创建带 IK 的新版本索引
- 绑定活动别名
- 查询始终使用活动别名
- 支持切换别名到新索引

当前 `deleteAll()` 的“删索引再重建”逻辑不再适合作为正式升级路径。

开发环境可以保留简化逻辑，但正式 IK 改造应新增独立索引管理能力。

### 6.3 查询逻辑

当前 `multi_match` 查询结构可继续保留：

- `title^3`
- `summary^2`
- `content`

接入 IK 后重点不是更换接口，而是让这些字段的 analyzer 生效。

高亮逻辑也可继续沿用现有 ES 原生 highlight 返回模式。

### 6.4 全量同步逻辑

全量同步需要具备“向指定目标索引构建”的能力，而不是默认只写当前索引。

建议：

- 创建新索引
- 全量同步写入新索引
- 验证通过后切换别名

### 6.5 增量同步逻辑

增量同步主逻辑保持不变，仍然走：

- Canal 事件
- 回源 MySQL
- 重建 ES 文档

但在索引切换期间，要明确“增量写到哪个索引”。

## 7. 切换与回滚策略

### 7.1 推荐切换流程

推荐流程如下：

1. 确认 ES 节点已安装 IK 插件
2. 创建新版本索引及别名准备
3. 对新索引执行全量同步
4. 验证新索引搜索效果
5. 切换活动别名到新索引
6. 保留旧索引一段时间作为回滚点

### 7.2 切换窗口问题

这里有一个重要点：

全量构建新索引期间，线上仍可能持续产生增量变更。

如果不处理这段时间的增量事件，就可能出现：

- 新索引数据落后于旧索引
- 别名切换后短时间读到旧数据

因此，正式线上方案中需要补一层“切换窗口补偿”策略。

### 7.3 本轮默认结论

本次方案先明确：

- IK 接入必须走版本化索引
- 别名切换是默认路线
- 切换窗口增量补偿属于下一层实现重点

如果只是开发环境或演示环境，可以先接受：

- 全量构建完成后直接切换

但若用于线上，必须补充切换窗口内的增量处理设计。

### 7.4 回滚方式

若新索引验证失败：

- 将别名重新指回旧索引
- 保留新索引用于问题排查
- 不修改 MySQL 与业务写路径

这样回滚成本最低，也不会破坏现有业务主链路。

---

## 8. 风险与兼容性

### 8.1 插件依赖风险

IK 是 ES 节点插件，如果：

- 节点版本不匹配
- 只有部分节点安装
- 安装后未重启

都可能导致：

- 索引创建失败
- 分片分配异常
- 查询报错

### 8.2 索引不兼容风险

接入 IK 本质上是索引结构升级，不是一次普通字段变更。

因此必须避免：

- 在原索引上直接改 analyzer
- 在未重建数据时切换搜索行为

### 8.3 切换窗口一致性风险

如果新索引全量构建完成前又有新的 MySQL 变更，而这些变更没有同步到新索引，则会产生短暂数据不一致。

这是本次方案中最需要重点关注的上线风险。

### 8.4 接口兼容性

本次改造不计划修改：

- `/search` 请求参数
- `/search` 响应结构
- 业务写接口

因此对外接口基本保持兼容，影响主要集中在底层索引结构与搜索效果。

## 9. 验证方案

### 9.1 环境验证

验证点：

- 所有 ES 节点已安装 IK
- 插件版本与 ES 版本匹配
- 新索引可成功创建

### 9.2 索引验证

验证点：

- `title` / `summary` / `content` 的 analyzer 设置生效
- 索引 mapping 和 settings 符合预期
- 别名正确指向目标索引

### 9.3 搜索行为验证

验证点：

- 中文标题搜索可命中
- 中文短语搜索召回提升
- 多词组合检索结果更符合预期
- 标题权重仍高于摘要和正文
- 高亮在中文搜索下仍能正常返回

### 9.4 回归验证

验证点：

- 分类过滤不受影响
- 标签过滤不受影响
- 作者过滤不受影响
- 状态过滤不受影响
- 时间范围过滤不受影响
- 无关键字排序行为不受影响

### 9.5 别名切换与回滚验证

验证点：

- 切换别名后查询能读到新索引
- 切回别名后查询能恢复旧索引结果
- 切换过程不需要修改业务代码配置

---

## 10. 建议实施顺序

建议按以下顺序推进正式开发：

1. 先完成 ES 节点 IK 插件安装与环境验证
2. 设计并实现版本化索引与别名管理能力
3. 为文章索引显式定义带 IK 的 settings 和 mappings
4. 改造全量同步，使其能向指定目标索引构建
5. 完成新索引全量构建与搜索验证
6. 增加别名切换与回滚能力
7. 最后补切换窗口内增量补偿策略

---

## 11. 结论

IK 分词接入不是单纯“给字段改个注解”，而是一次完整的搜索索引升级工作。

本次方案的核心结论是：

- IK 插件安装是前置条件
- 索引升级必须走版本化重建
- 查询接口可保持不变
- 高亮与多字段权重策略可以继承现有实现
- 真正的上线难点不在查询语法，而在索引切换和增量一致性保障

因此，后续正式开发应以“插件准备 -> 索引版本化 -> 全量构建 -> 别名切换 -> 一致性补偿”为主线推进，而不是直接在当前索引上硬改 mapping。
